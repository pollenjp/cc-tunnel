## 背景

docker_gce provider 導入以降、ログ出力源は次のように分散している。

| レイヤ | 出力源 | 取り込み状況 |
|---|---|---|
| cc-tunnel (Cloud Run) | stdout JSON (slog) | Cloud Run 自動取り込み済み |
| container-manager (VM 常駐) | stdout JSON (slog) | **VM 内の `docker logs` 内に留まり Cloud Logging に届かない** |
| cc-remote-agent (per-session) | stdout/stderr | 同上 |
| VM systemd / kernel | journald | 取り込みなし |

VM SA には `roles/logging.logWriter` / `roles/monitoring.metricWriter` が付与済み (`terraform/modules/cc-tunnel/gce_vm_sa.tf:22-30`) だが、Packer image (Ubuntu 24.04 + docker-ce) には **Ops Agent 未インストール**、Docker daemon の log driver も `json-file` のため、実際にはどのログも GCP 側で観測できていない。

会話単位の障害解析・性能調査をするには「Cloud Run の cc-tunnel 側ログ」と「VM 上 container-manager / cc-remote-agent のログ」を `conversation_id` で串刺し検索できる必要がある。現状はそれが不可能。

---

## 決定事項

**全レイヤのログを Cloud Logging に集約する。** 取り込み手段はレイヤ別に最小コストの仕組みを選ぶ。

### 1. レイヤ別の取り込み方法

| レイヤ | 手段 | 補足 |
|---|---|---|
| cc-tunnel (Cloud Run) | 既存 stdout JSON (Cloud Run 自動) | slog の属性名を Cloud Logging 規約 (`severity` / `message` / `timestamp`) に揃える |
| container-manager (VM 常駐コンテナ) | Docker log driver = `gcplogs` | systemd unit の `docker run` に `--log-driver=gcplogs --log-opt labels=component,version` を付ける |
| cc-remote-agent (per-session) | Docker log driver = `gcplogs` | container-manager が `RunAgent` 時に `HostConfig.LogConfig` で強制 |
| VM systemd / kernel / dockerd 本体 | Ops Agent (`google-cloud-ops-agent`) | Packer image に apt インストールを追加 |

### 2. `gcplogs` を選んだ理由

- Docker 公式の log driver で、stdout/stderr を Cloud Logging API に直接送る。
- Container の Docker labels をそのまま Cloud Logging の `labels` フィールドに反映できる (`--log-opt labels=<csv>`)。
- VM の Instance ID / Zone は自動的に `resource.labels` に入る。
- Ops Agent + fluent-bit で `/var/lib/docker/containers/*/*.log` を tail する方式に比べ:
  - 設定が大幅に簡潔。
  - Docker labels → Cloud Logging labels の自動マッピングが効く (fluent-bit では filter で書き換える必要がある)。

Ops Agent は systemd / kernel / dockerd 本体のログ用に限定して入れる。

### 3. JSON ログ規約 (Go 側)

cc-tunnel と container-manager の `slog` 初期化を共通化し、Cloud Logging が認識する構造化フィールド名に合わせる:

| slog 標準キー | Cloud Logging 標準キー | 変換 |
|---|---|---|
| `level` | `severity` | `INFO` / `WARN` / `ERROR` などをそのまま大文字で渡せばよい |
| `msg` | `message` | リネーム |
| `time` | `timestamp` | リネーム |

`gcplogs` driver は JSON ペイロードを Cloud Logging の `jsonPayload` にそのまま入れる。`severity` フィールドがトップレベルにあると Cloud Logging が severity を自動認識するため、検索とアラート設定が容易になる。

実装は `apps/cc-tunnel/internal/logging` パッケージに共通ヘルパを置き、container-manager 側はそれを import する (両アプリは同一 monorepo)。

### 4. ラベル設計 (横断検索の鍵)

全層で同名キーを使うことで Cloud Logging クエリ `labels."conversation_id"="..."` で串刺しできる:

| キー | 値の例 | 設定箇所 |
|---|---|---|
| `conversation_id` | `c8f1...` | cc-tunnel slog 属性、container-manager の `CreateAgent` リクエスト → Docker container label → `gcplogs labels` |
| `vm_instance_id` | DB UUID | 同上 |
| `gce_instance_name` | `cc-tunnel-a1b2c3d4` | gcplogs が自動付与する `compute.googleapis.com/resource_name` で代用可能だが、DB 上の `vm_instances.id` と突き合わせるため別途付ける |
| `component` | `cc-tunnel` / `container-manager` / `cc-remote-agent` | アプリ初期化時に固定 |

ラベルの伝搬パス:

```
cc-tunnel (slog attr: conversation_id, vm_instance_id)
  ↓ HTTP POST /v1/agents (request body: labels[])
container-manager (HTTP handler) 
  ↓ RunAgent (Docker container labels & LogConfig.Config["labels"])
Docker daemon
  ↓ stdout/stderr + gcplogs driver
Cloud Logging (labels.conversation_id, labels.vm_instance_id)
```

### 5. Docker daemon 側のフォールバック

`gcplogs` driver 利用時は VM 内ファイルにログが溜まらないため、デバッグ目的の `docker logs` も空になる。完全に Cloud Logging に依存する形になる。

ただし `gcplogs` が一時的に Cloud Logging API へ到達できない場合に備え、`/etc/docker/daemon.json` で:

```json
{
  "log-driver": "gcplogs",
  "log-opts": {
    "gcp-log-cmd": "true"
  }
}
```

としつつ、後段の改善案として `gcplogs-cache=true` への切り替え余地を残す (現状は不採用。理由: cache はメモリ上で吸収するだけで、Cloud Logging API が長時間ダウンするとロストする。Cloud Logging API の SLA を信頼する判断)。

### 6. 監視・アラート

ログベース指標 (Cloud Monitoring) として以下を整備する (本 ADR スコープ外、後続 PR):

- `cc-tunnel` の `severity>=ERROR` カウント (provider 別)
- container-manager の `ImagePull` / `ContainerCreate` 失敗率
- VM 起動から `IsReady=true` までの所要時間 (Cloud Logging 上の JSON フィールドから抽出)

### 7. コスト/保持

- デフォルト 30 日保持で本 ADR スコープでは十分。
- 長期保存が必要なログは後続 PR で BigQuery sink を追加 (例: 監査・課金関連)。

## 却下した代替案

### 却下: Ops Agent + fluent-bit で `/var/lib/docker/containers/*/*.log` を tail する案

- メリット: Docker daemon の log driver を `json-file` のまま据え置ける、`docker logs` がそのまま使える。
- 却下理由:
  - fluent-bit のラベル抽出設定 (Docker container label を Cloud Logging labels に持ち上げる) が複雑。Ops Agent の現行スキーマでは Docker labels の reshape は素直に書けず、`lua` フィルタが必要。
  - container-manager / cc-remote-agent / 将来追加されるサイドカーで毎回 fluent-bit config をいじる必要がある。
  - `gcplogs` は Docker daemon ネイティブで、運用負荷が一段低い。

### 却下: cc-remote-agent 側で SDK 経由で Cloud Logging API を直接叩く案

- メリット: 構造化ログをアプリ側で完全制御できる。
- 却下理由:
  - cc-remote-agent の SA に Logging 権限を付ける必要があり、`2026-05-06_container_manager_on_vm` で確立した「VM レイヤだけが GCP API を叩く」という権限分離方針に反する。
  - アプリ側で SDK 依存とリトライ・バッチ処理を抱え、責務が肥大化する。

### 却下: 構造化ログを送らず、`docker logs` + SSH で都度確認する案

- メリット: 設定変更ゼロ。
- 却下理由:
  - そもそも VM には SSH バイナリが入っていない (`COS 相当` 方針、ただし Ubuntu に変わったので技術的には可能)。
  - 短命 VM では VM 削除後にログがロストする。会話の事後解析が不可能。
  - 複数 VM をまたいだ検索ができない。

## セキュリティ考慮事項

- `gcplogs` driver が使うトークンは VM SA の metadata server 由来。container-manager と同じ経路で問題なし。
- Logs Explorer に流れるログには `ANTHROPIC_API_KEY` などの秘匿値を含めないこと。slog 属性名に注意 (cc-remote-agent からの stream-json は SSE に流れるためログには載せていない — 維持する)。
- container-manager の HTTP API に渡される `labels` は VM SA 経由で Cloud Logging に格納されるため、外部入力をそのまま流すと PII / API キーが漏れる可能性がある。**cc-tunnel が流す `labels` のキー集合は `conversation_id`, `vm_instance_id`, `component` のホワイトリストに限定する** (handler 側で reject はしないが cc-tunnel 側で生成のみ)。
