## 背景

docker_gce provider では、`active_containers` カウンタを根拠に DB
内でアイドル判定を行い、`IdleChecker` / `VMScaler` という 2 つの
in-process goroutine が定期的に `CleanupOrphans` / `ReconcileVMs` を
呼び出してアイドル VM を GCE API で削除する設計になっていた
(`internal/provider/dockergce/{idle_checker.go,vmscaler.go}`)。

この設計には次の 2 系統の欠陥がある。

### 1. 真実は VM 側の Docker daemon にしかない

`active_containers` は cc-tunnel の DB に書かれる **意図** の値であり、
実際に VM 上で agent コンテナが残っているかは別問題である。次のような
ケースで DB と VM の状態がドリフトする:

- cc-tunnel が異常終了して `active_containers--` を発行できなかった
- `docker rm` を外部 (運用者・systemd) から行った
- VM 上の container-manager が再起動して docker daemon の状態を再評価した
- 何らかの理由で cc-remote-agent がクラッシュした

DB を権威ソースにすると、利用中の VM を削除する／使われていない VM を
残し続けるという誤判定が起きる。

### 2. Cloud Run 上では in-process ticker が走らない

cc-tunnel は Cloud Run 上で動作する。Cloud Run は以下の挙動を持つ:

- **min instances=0** の場合、リクエストが無い間はインスタンスを停止する
  → goroutine そのものが存在しない
- リクエスト処理間 (request-period 外) は CPU を **throttle** する
  → ticker の発火が遅延・欠落する
- インスタンスが scale-in されると、たまたまその instance が抱えていた
  「最後の reap がいつだったか」の in-memory state が消える

**reap が最も必要なタイミング (= ユーザーが居ない静寂時間)** こそ、
in-process goroutine が動かないタイミングに一致する。つまり in-process
VMScaler は構造的に Cloud Run と相性が悪い。

### Step A (済) で既に対処したこと

PR #78 の途中までで以下を入れた:

- `vm_instances.zero_agents_since` カラム追加 (DB 駆動の意図値ではなく、
  container-manager から観測した実測値ゼロが続いている開始時刻)
- `DockerGCEProvider.ReconcileVMs` 実装: 全 running VM について
  container-manager に `GET /v1/agents` し、ゼロ継続が `ZeroAgentsTimeout`
  を超えた VM を GCE API で削除
- `VMReconcileInterval` のデフォルトを **0 (= goroutine を起動しない)** に
  変更し、production の Cloud Run では in-process ticker を使わないことを
  明示

`ReconcileVMs` 自体は信頼できる権威ソース (VM の Docker daemon) を
参照しているので残す。問題は **それを誰がどう周期実行するか** に変わる。

## 検討した選択肢

| # | 方式 | 周期 | Cloud Run 制約 | 障害耐性 | 運用コスト |
|---|---|---|---|---|---|
| 1 | Cloud Scheduler → cc-tunnel HTTP `/internal/reconcile-vms` (OIDC) | 6h | 影響なし | scheduler / Cloud Run / DB / VM 全部健全である必要あり (鎖が長い) | 最小 |
| 2 | Cloud Run Jobs (定期実行ジョブ) | 任意 | 影響なし | 1 と類似 | コンテナ追加・別 IAM・別ログ系統で運用増 |
| 3 | container-manager 自己終了 (VM 上 goroutine が自分の docker daemon を観測 → `instances.delete(self)`) | 10min | **影響なし** (VM 上で常時稼働) | VM が単独で完結。container-manager さえ生きていれば動く | self-delete IAM 設計が必要 |
| 4 | Cloud Run min-instances=1 にして in-process ticker を残す | 既存 | 部分的に解決 (それでもインスタンス入れ替えで state は消える) | scale-out 時に状態整合が崩れる | 常時 1 インスタンス分の固定費 |
| 5 | DB の `pg_cron` や Cloud SQL 側スケジューラ | 任意 | 影響なし | DB に GCE API 呼ぶロジックを置くのは責務違反 | 大きい (DB に compute SA 権限要) |

### なぜ 1 と 3 の組合せか

- **3 を主**にする理由:
  - 周期を細かく (10min) しても Cloud Scheduler のジョブ数や料金に影響しない (VM 上は無料)
  - VM 内で完結するため、cc-tunnel / scheduler / 中間ネットワークいずれかが壊れていても reap が走る
  - 「使い終わった瞬間に消える」感覚に最も近い (10min 後)
  - container-manager は既に VM 上に常駐しているので追加コンポーネントが要らない
- **1 を保険**にする理由:
  - container-manager 自身がクラッシュ・OOM・systemd 障害で死んでいる VM は 3 では絶対消せない
  - **「孤児 VM が無限に課金され続ける」のは事業上最悪のシナリオ**なので、外部から強制的にスイープする経路を必ず持ちたい
  - 6h は「container-manager が落ちている異常事態を 6h 以内に検知して掃除する」という SLA として十分・かつ Cloud Scheduler 課金もごく少額
  - DB 上 running なのに GCE 上に存在しない孤児行の掃除もここで併発できる
- **4 は不採用**: 固定費が継続する。in-process state が scale-out で壊れる根本問題が残る
- **2 は不採用**: 1 とほぼ等価な機能で運用面のコストだけ増える
- **5 は不採用**: DB に compute SA を持たせない方針

## 決定事項

VM reap を以下の 2 系統に分離する。`active_containers` は **スケジューラ
向けヒント** として残し (`getOrCreateVMInstance` で容量見積りに使う)、
**削除判定の権威ソースからは外す**。

| 経路 | 主従 | 周期 | 起動者 | 呼び先 | 認証 |
|---|---|---|---|---|---|
| container-manager 自己終了 | 主 | 10min | 各 VM 上の container-manager goroutine | `compute.instances.delete(self)` | VM SA の self-delete |
| Cloud Scheduler → cc-tunnel HTTP | 保険 | 6h | Cloud Scheduler (Google managed) | `POST /internal/reconcile-vms` → 既存 `ReconcileVMs` | OIDC ID token (scheduler SA → Cloud Run audience) |

### container-manager 自己終了 (主)

各 VM 上の container-manager に self-reaper goroutine を追加する。

- `SELF_REAP_INTERVAL_SECONDS` (default `60`) 毎に自身の Docker daemon に
  `label=component=cc-remote-agent` でコンテナ一覧を取得
- count > 0 → in-memory `zeroSince = nil`
- count == 0 かつ `zeroSince == nil` → `zeroSince = now`
- count == 0 かつ `zeroSince < now - SELF_REAP_TIMEOUT (10min)` →
  GCE metadata server から自身の `instance name` / `zone` / `project_id` を
  取得し、`compute.instances.delete(self)` を呼ぶ
- API 呼び出し成功後は systemd が container-manager / agent コンテナごと
  停止するのを待つ (能動的な OS shutdown は不要、GCE 側がやる)

**IAM**: VM SA に対し `compute.instances.delete` を「自身の instance のみ」
許可する。プロジェクト全体に `compute.instanceAdmin.v1` を与えると別の
セッションの VM も消せてしまうので、custom role + IAM Conditions
(`resource.name == ...`) か、最小権限 `roles/compute.instanceAdmin` を
**当該 VM リソースに対してのみ** 個別付与する形にする。実装スコープは
PR #78 内で決める。

### Cloud Scheduler → cc-tunnel HTTP (保険)

- cron `0 */6 * * *` (6 時間ごと)
- target: `POST https://<cloud-run-url>/internal/reconcile-vms`
- 認証: OIDC ID token を `Authorization: Bearer <token>` で付与
  - issuer: `https://accounts.google.com`
  - audience: Cloud Run service の URL (default audience)
  - subject (email): scheduler 専用 SA
- cc-tunnel 側は `google.golang.org/api/idtoken.Validate` で検証し、
  許可された SA email リスト (env で指定) と一致しなければ 403
- handler は既存の `DockerGCEProvider.ReconcileVMs(ctx)` を呼ぶだけ
  (OpenAPI には乗せず、main.go の mux に直接登録する)

**冪等性**: `ReconcileVMs` は既に冪等 (`zero_agents_since` を更新するか、
しきい値超えで削除するか、何もしない)。Cloud Scheduler のリトライで
重複起動しても問題ない。

### Terraform

- `cc-tunnel/scheduler.tf` を新規追加:
  - `google_service_account.scheduler_sa`
  - `google_cloud_scheduler_job.reconcile_vms` (HTTP target, OIDC)
  - `google_cloud_run_service_iam_member.scheduler_invoker` (Cloud Run 側 invoker 権限)
- `cc-tunnel/gce_vm_sa.tf` を更新: VM SA に self-delete IAM を付与
- `google_project_service` で `cloudscheduler.googleapis.com` を有効化

## 影響範囲

### Step A (`b115bc0` 済)
- `VMReconcileInterval` default = 0 にして in-process goroutine を停止
- `docs/docker-gce-design.md §5.2` 改訂

### Step B (本 ADR でカバー、未実装)
- cc-tunnel に `POST /internal/reconcile-vms` ハンドラ追加 + OIDC 検証
- Terraform: Cloud Scheduler job / scheduler SA / IAM / API enablement

### Step C (本 ADR でカバー、未実装)
- container-manager に self-reaper パッケージ追加
- container-manager の GCE compute client + metadata server 利用
- VM SA に self-delete IAM
- Packer image 側で新しい環境変数 (`SELF_REAP_TIMEOUT_SECONDS` 等) を
  systemd unit に渡す

### 既存テスト
- `TestVMScaler_Started` は `VMReconcileInterval` を明示的に与える形に
  既に修正済み (Step A の commit に含む)
- in-process goroutine を前提とした他のテストは無し
- `ReconcileVMs` の単体テストは `reconcile_vms_test.go` に既存

### 運用への影響
- 既存環境で `GCE_VM_RECONCILE_INTERVAL_SECONDS` を明示的に設定していた
  場合は引き続き有効 (後方互換)
- 設定していなければ default 0 = goroutine 起動なし → Step C デプロイ前は
  reap が動かない期間が発生する。Step A → B → C を短期間で連続デプロイする
- Cloud Scheduler の月額は VM 1〜2 円未満 (6h おき = 月 ~120 起動)

## やらないこと / 将来課題

- **DB スキーマからの `active_containers` 削除**: ヒントとしては残すので
  今は触らない。完全に不要と判断できた段階で別 ADR で扱う
- **container-manager の HA**: VM ごとに 1 つ動いていれば足りる。複数台
  跨ぎのリーダー選出は不要
- **Cloud Scheduler から container-manager を直接叩く**: VPC 経由でないと
  container-manager に到達できないので採用しない (Cloud Scheduler は VPC
  に入れない)。cc-tunnel が VPC Connector 経由で仲介する現状を維持する
- **緊急停止 API**: 「今すぐ全 VM を消したい」という運用 API は別途
  考える。今回の reap は時間ベースのみ
