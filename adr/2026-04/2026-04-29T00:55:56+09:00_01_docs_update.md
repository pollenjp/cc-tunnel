# ADR: cc-tunnel ドキュメント更新

## 背景・経緯

cc-tunnel の実装が進むにつれ、docs/ 配下の各種ドキュメントが実態と乖離していた。
特に以下の点でドキュメントと実装の不一致が発生していた：

- DockerGCEProvider が完全実装済み（モックではなく本格実装）になっていた
- 新規 DB テーブル（vm_instances / session_endpoints）がマイグレーション 005/006 で追加されていた
- IdleChecker と VMScaler が別 goroutine として実装されていた
- Postgres イメージが 17 から 18-alpine に変更されていた
- compose.yaml で cc-tunnel がデフォルトサービス、frontend が profile: full に変更されていた
- フロントエンドの React バージョンが 18 → 19 に更新されていた

## 変更の概要

足軽1号（subtask_cctdoc_001a）が docs/*.md を更新し、本タスク（subtask_cctdoc_001b）で
docs/plantuml/*.puml の更新と ADR 作成を行った。

## TDDアプローチ

本タスクはドキュメント更新であり、TDD の直接適用はなかった。
ただし、ソースコード（provider.go, idle_checker.go, vmscaler.go）を参照して
変更ノートの記述と実装の整合性を確認してから図を更新した。

具体的には：
1. 変更ノート（docs/_change_notes.md）を読み、変更点を把握
2. 各 PlantUML ファイルの現状を読み、実装との差分を特定
3. provider.go で IdleChecker と VMScaler の実際の動作を確認
4. 差分がある箇所のみ更新（過剰な変更を避けた）

## 今後の注意点

- VMScaler は `vmscaler.go` として実装済みだが、現状の `NewDockerGCEProvider` コンストラクタでは VMScaler の自動起動はしていない（`IdleCheckInterval` が設定された場合のみ IdleChecker が起動）。将来 VMScaler を使用する場合はプロバイダ設定・コンストラクタの更新が必要。
- PlantUML ファイルは Java/plantuml.jar なしでテキスト編集のみで更新可能。レンダリング確認は CI または別途 jar を使用すること。
- 今後の実装変更時は docs/_change_notes.md 形式の引き継ぎノートを活用し、足軽1号（*.md 更新）→足軽2号（*.puml 更新）の分業フローを維持すること。
