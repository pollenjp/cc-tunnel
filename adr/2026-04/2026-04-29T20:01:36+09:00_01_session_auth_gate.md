# ADR: セッション認証ゲート (Session Auth Gate)

- **Date**: 2026-04-29T20:01:36+09:00
- **Status**: Accepted
- **Related cmd**: `cmd_cctunnel_session_auth_gate`

---

## 背景

`/chat/:id` ルートは `<AppAuthGuard>` (JWT 検証) のみで保護されていたが、
3 つの欠落があった:

| # | 欠落 | 影響 |
|---|------|------|
| G1 | `GET /api/credentials/status` backend 未実装 | frontend が status チェック不可 |
| G2 | `App.tsx` で `<CredentialGuard>` が未適用 (dead code) | credential 不在でもチャット画面に入れてしまう |
| G3 | `sendMessage` の 401 redirect 未処理 | race condition 時に redirect 先へ遷移しない |

credentials が未登録・失効のままメッセージ送信が試みられると、
backend (`SendMessage`) が 401 を返すが frontend がそれを無視してしまう状態だった。

---

## 採用した解決策

### 1. `GET /api/credentials/status` 追加 (backend)

- `openapi.yaml` に `/credentials/status` パス・`CredentialsStatusResponse` スキーマを追加
- `gen.go` を `go generate` で再生成
- `handler.go` に `GetCredentialsStatus` 実装:
  - `credService == nil` → `{registered: true, isValid: true}` (no-auth / skip mode 互換)
  - `FetchAndDecrypt` 成功 → `{registered: true, isValid: true}`
  - `ErrNotFound` → `{registered: false, isValid: false}`
  - `ErrCredentialsInvalid` → `{registered: true, isValid: false}`
  - その他エラー → 500

### 2. `<CredentialGuard>` の `/chat/:id` への適用 (frontend)

- `ChatRouteWithCredGuard.tsx` (軽量 wrapper) を新規作成:
  `useParams()` で `id` を取り `<CredentialGuard conversationId={id}><ChatPage /></CredentialGuard>` を返す
- `App.tsx` の `/chat/:id` ルートを `<AppAuthGuard><ChatRouteWithCredGuard /></AppAuthGuard>` 構造に変更
- `/chat` (no id) は **CredentialGuard を適用しない** — 会話未作成のため status チェック不要

### 3. `sendMessage` の 401 redirect 処理 (frontend)

- `client.ts` の `sendMessage` に reactive fallback を追加:
  `!res.ok` かつ status=401 のとき response body の `redirect` フィールドを読み `window.location.assign(redirect)` で navigate
- これは race condition (status check 通過後に他端末で credential 削除) への第二ゲート

---

## 不採用案

### 案A: 401 reactive fallback のみ

`sendMessage` の 401 処理だけを実装し、CredentialGuard + status API を省く案。

**不採用理由**:
- ユーザーが `/chat/:id` を開いた瞬間に credential 状態を判定できない
- チャット画面が一瞬表示された後にリダイレクトされる UX が劣悪
- `<CredentialGuard>` は既に実装済みだが dead code のままになり技術的負債が残る

### 案B: middleware redirect (backend)

AppAuthGuard 相当のミドルウェアで credential を毎回確認し、不正時は HTTP 302 を返す案。

**不採用理由**:
- SPA 構成において HTTP 302 はブラウザの fetch API で follow されてしまい、
  frontend が redirect 先を制御できない
- 全 API リクエストに credential チェックのオーバーヘッドが発生する

---

## 実装上の決定事項

| 決定 | 内容 | 理由 |
|------|------|------|
| `/chat/:id` のみ CredentialGuard 適用 | `/chat` (no id) には適用しない | 会話作成前は credential 不要; conversationId がないと status チェックの意味が薄い |
| `credService == nil` 時は常に pass | `{registered: true, isValid: true}` を返す | no-auth mode / テスト環境との後方互換 |
| status エラー時は fail-open | `CredentialGuard` は status API 呼び出し失敗時に children を表示 | `SendMessage` が second-line gate になるため、UX 優先で pass させる |
| CredentialGuard は App.tsx ではなく wrapper component 経由 | `ChatRouteWithCredGuard.tsx` に委譲 | `useParams` を App.tsx レベルで呼ぶとルート構造が複雑になるため |
