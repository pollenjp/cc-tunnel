# ADR: xterm.js PTY レンダリング統合

**日時**: 2026-04-29T20:46:14+09:00
**ステータス**: 採用

---

## 背景

`apps/frontend/src/pages/CredentialsLoginPage.tsx` は、PTY 出力を独自のポーリング実装（`getAuthOutput` → `atob` → `replace(/\r/g, '')` → `div` 描画）で行っていた。

この実装では ANSI エスケープシーケンス（`\x1b[...m` 等）がプレーンテキストとして表示され、認証フロー中の PTY 出力が文字化けする問題が発生していた。

一方、`apps/frontend/src/components/AuthTerminal.tsx` は `@xterm/xterm` を使って同じ PTY 出力を正確にレンダリングする完成済みコンポーネントが存在しており、`AgentSettingsPage` で稼働中だった。

---

## 採用案: AuthTerminal 再利用 + onTextOutput callback

`AuthTerminal` コンポーネントに `onTextOutput?: (text: string) => void` callback prop を追加し、`CredentialsLoginPage` の独自 PTY 描画ロジックを `<AuthTerminal>` で置換する。

- `pollOutput` 内で base64 デコード後の `binary` を `xtermRef.current?.write(bytes)` に渡した直後に `onTextOutput?.(binary)` を呼ぶ
- `CredentialsLoginPage` 側では `fullOutputRef` に累積し、`/Login successful|Logged in|authentication successful/i` マッチで `doFinalize()` を呼ぶ既存ロジックを維持する
- callback は optional のため既存利用（`AgentSettingsPage`、`AuthGuard`）への影響ゼロ

**採用理由**:
- 既存の xterm.js 資産を再利用し、ANSI レンダリングの問題を根本解決できる
- ポーリング系統が AuthTerminal に一本化され、二重ポーリングが発生しない
- 変更箇所が最小限で、テスト負荷も低い

---

## 不採用案

### 案X: ANSI strip のみ

`replace(/\x1b\[[0-9;]*m/g, '')` 等で ANSI エスケープを除去する。

**却下理由**: 殿の方針（ANSI エスケープを strip しない、xterm.js でカラー含めて忠実にレンダリングする）に明示的に反する。色情報が失われ、カーソル制御も無効になる。

### 案Y: 新コンポーネント `CredentialPtyTerminal.tsx` を新規作成

`CredentialsLoginPage` 専用の xterm.js コンポーネントを別途作成する。

**却下理由**: `AuthTerminal` とほぼ同コードを重複作成することになり、保守コストが増大する。採用案より変更が多く、価値が乏しい。

### 案Z: CredentialsLoginPage 内で xterm を直接 instantiate

`CredentialsLoginPage` 内で `new Terminal()` を直接呼び出して PTY を描画する。

**却下理由**: `AuthTerminal` が既に存在するのに同等ロジックを重複させることになる。改修箇所が増え、テスト負荷も増大する。

### 案A: 二重ポーリング（AuthTerminal + 既存ポーリング併存）

既存の独自ポーリングを残したまま `AuthTerminal` を追加する。

**却下理由**: `/auth/output` API への二重アクセスが発生し、cursor 管理が競合する。
