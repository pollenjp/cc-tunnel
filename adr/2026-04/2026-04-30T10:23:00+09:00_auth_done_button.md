# ADR: 認証「完了」ボタン追加 — 二重トリガー設計

- **日時**: 2026-04-30T10:23:00+09:00
- **対象**: `apps/frontend/src/pages/CredentialsLoginPage.tsx`
- **タスク**: subtask_adbc_001b

---

## 背景

### 現在の自動検知タイミング

`CredentialsLoginPage` は PTY 出力（SSE 経由）を `onTextOutput` callback で受信し、
`/Login successful|Logged in|authentication successful/i` にマッチした瞬間に自動で
`doFinalize()` を呼び出す。これにより `POST /api/credentials/relogin/finalize` が実行され、
`~/.claude/.credentials.json` が AES-256-GCM で暗号化されて PostgreSQL へ UPSERT される。

### 検知失敗リスク

- claude CLI の出力文字列が将来変更された場合、regex がマッチしない
- TUI レンダリング順序によって文字列が分断され、マッチが遅延・失敗する場合がある
- 認証が成功しているのに finalize されず、ユーザーが何分待っても次に進めない状態に陥る

### TUI の性質

`claude /auth` は対話型 TUI でセッションが明示的に終了しない。ユーザーが認証操作を
完了したタイミングで明示的に flush できる手段がなかった。

---

## 採用案: 既存 API 再利用 + 二重トリガー（自動検知継続 + 明示ボタン）

`phase === 'pty'` 時に「完了」ボタンを追加し、押下時に既存の `doFinalize()` を呼び出す。
自動検知ロジックは**維持**する。

- **新 API・backend 改修不要**: 既存 `POST /api/credentials/relogin/finalize` を共有
- **多重実行防止**: 既存 `finalizedRef.current` ガードがそのまま有効
- **失敗時リトライ余地**: catch ブロックで `finalizedRef.current = false` にリセット
- **エラーメッセージ**: `credentials_not_ready` 含むエラーは
  「認証が完了していません。画面の指示に従って認証を完了してください」と表示

---

## 不採用案と根拠

| 案 | 不採用理由 |
|----|-----------|
| 新 API endpoint 追加 | 既存 `PostReloginFinalize` で全て対応可能。over-engineering になる |
| 自動検知のみ | 検知失敗時にユーザーが詰む。救済手段がない |
| 完了ボタンのみ（自動検知廃止） | 既存の自動完了 UX を破壊する regression リスクがある |
| error フェーズでもボタン再表示 | UX が複雑になる。error 時はリロード案内で十分 |

---

## 実装の決定事項

- **改修範囲**: フロントエンド 2 ファイルのみ（tsx + テスト）
- **backend 不要**: 既存 endpoint を再利用するため
- **openapi.yaml 不要**: 新規 path 追加なし
- **ボタン表示条件**: `phase === 'pty'` のときのみ（starting/finalizing/done/error 中は非表示）
- **キャンセルボタン**: flex コンテナ内に移動し完了ボタンと横並び配置
