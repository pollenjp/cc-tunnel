# ADR: SSE PTY ストリーミング — ANSI カラー品質劣化の根本解決

**日時**: 2026-04-30T01:26:09+09:00
**ステータス**: 採用

---

## 背景・根本原因

### 殿の認識と実態のギャップ

殿の初期認識は「DB 経由の auth 入出力が品質劣化の根本原因」であったが、軍師の調査で実態が判明した。

`apps/cc-tunnel/internal/db/migrations/*.sql` に auth 関連テーブルは存在しない。PTY 出力は **cc-remote-agent の in-memory `outputBuf`** で管理されており、DB 保存は行われていなかった。

### 真の根本原因 3 点

| # | 根因 ID | 内容 | 場所 |
|---|---------|------|------|
| R1 | **stripANSI** | PTY 出力に `stripANSI()` を適用し ANSI エスケープを破壊していた | `apps/cc-remote-agent/internal/auth/manager.go` (旧 line 145) |
| R2 | **累積バッファ + polling** | `outputBuf` に累積して 250ms 間隔でクライアントが pull する設計。遅延が蓄積し TUI の応答性が悪かった | `auth/manager.go:41,161-174` + `frontend AuthTerminal.tsx:65` |
| R3 | **設計矛盾** | frontend が xterm.js でカラー描画しようとしても、backend で ANSI を剥ぎ取っているため永遠に色が出ない | 設計上の矛盾 |

`subtask_xtermjs_001b` で frontend の xterm 化は完了していたが、backend の stripANSI が ANSI エスケープを破壊していたため、カラー描画は実現不能な状態だった。

### 殿の裁定 (cmd_cctunnel_ansi_strip_pty_output キャンセル → SSE 根本変更)

- `cmd_cctunnel_ansi_strip_pty_output`（stripANSI 削除のみ）を **キャンセル**
- SSE (Server-Sent Events) + EventSource による根本的な再設計を採用
- R1 の stripANSI 削除、R2 の累積バッファ廃止・polling 廃止、R3 の設計矛盾解消を同時達成

### 参照 commit 575ee04 は存在しなかった

殿が言及した commit `575ee04` は `git log --all --oneline | grep 575ee04` でヒットしない（別環境または記憶違い）。当該 commit によるロールバックは不可能と確認した。

## 採用案: SSE (EventSource) + stripANSI 削除

### 変更概要

| 層 | 変更内容 |
|---|---------|
| **cc-remote-agent** | `auth/ansi.go` + `ansi_test.go` 削除。`manager.go` から `outputBuf` field・`GetOutput` メソッド・`stripANSI` 呼出を削除。fan-out subscribers map 追加。`Subscribe(ctx)` メソッド追加。`BroadcastForTest` テストヘルパー追加 |
| **cc-remote-agent API** | `/auth/output` (polling) を廃止し `/auth/pty/stream` (SSE) を追加。`/auth/input` を `/auth/pty/input` にリネーム |
| **cc-tunnel API** | `GetAuthOutput` を廃止し `GetAuthPtyStream` (SSE 中継) を追加。`SubmitAuthInput` を `SubmitAuthPtyInput` に対応 |
| **openapi.yaml** | `/auth/output` 削除。`/auth/pty/stream` (GET, SSE) 追加。`/auth/input` → `/auth/pty/input` リネーム |
| **frontend** | `AuthTerminal.tsx` の `setInterval` polling を `EventSource` に置換。`api/client.ts` から `getAuthOutput` 削除・`submitAuthPtyInput` 追加 |

### 実装の決定事項

| 項目 | 決定内容 |
|------|---------|
| fan-out 実装 | `subscribers map[chan<- []byte]struct{}` + `subscribersMu sync.Mutex` |
| チャネルバッファサイズ | 64 (溢れた場合は chunk を drop してログ出力) |
| base64 エンコード | `base64.StdEncoding.EncodeToString` で SSE `data:` フィールドに格納 |
| heartbeat | 30 秒ごとに SSE comment `: keepalive\n\n` を送信 (Cloud Run / nginx proxy の keepalive 対応) |
| 再接続時の履歴 | PTY 本体は AuthManager で維持。再接続時はキー入力で TUI が再描画するため許容範囲 |
| PTY 切断時 | subscriber context cancel により subscribers map から自動削除。PTY 本体は維持 |

## 不採用案

### WebSocket

**却下**: PTY stdin→stdout の単純な一方向ストリーミングに双方向通信は不要。接続確立コストや中継実装コストが SSE より高い。stdin 入力は引き続き `POST /auth/pty/input` で処理する。

### fetch + ReadableStream

**却下**: 自動再接続が EventSource ほど単純でない（手動実装が必要）。デコードのボイラープレートが増える。既存認証が cookie ベースで Bearer token が query string 不要のため EventSource で十分。

### stripANSI 削除のみ (cmd_cctunnel_ansi_strip_pty_output)

**却下**: R1 は解消されるが、R2 (250ms polling による遅延) が残存する。TUI のリアルタイム性が確保できない。殿の裁定によりキャンセル済み。

## 結果

- `mise run check` 全 PASS
- `apps/cc-remote-agent/internal/auth/manager.go` から `stripANSI` 呼出が消滅 (grep でゼロヒット)
- `apps/cc-remote-agent/internal/auth/ansi.go` が削除済み
- `/auth/pty/stream` (SSE) と `/auth/pty/input` (POST) が openapi.yaml に存在
- `apps/frontend/src/api/client.ts` から `getAuthOutput` が消滅
- `apps/frontend/src/components/AuthTerminal.tsx` の `setInterval` polling が消滅し `EventSource` で SSE 受信
- E2E テスト `TestAuthSSEPtyStream_E2E_ANSINotStripped` / `TestAuthSSEPtyStream_E2E_HeartbeatSent` が PASS
- 一時ガイド `docs/_sse_pty_stream_impl_guide.md` 削除済み
